# Agentic Chat 与自主编辑执行计划

## 背景

当前右侧 AI 面板仍然更接近“单次选区改写工具”，而不是可持续协作的 AI Chat。用户已经明确需要把它升级为更像真实 AI coding / writing assistant 的交互与执行模型，包括多轮对话、自动滚动、无选区任务处理、工具调用以及 agent 自主循环。

这意味着右侧栏的目标需要从“改写触发器”升级为“可持续协作的文档代理工作台”。

## 当前目标

1. 右侧 AI Chat 输入框支持按回车直接发送
2. AI Chat 支持多轮会话、消息历史、流式消息更新和自动滚动
3. 没有选区时也可以发起任务，由 AI Agent 基于指令自主决定如何读写当前文档
4. 为 AI Agent 提供 `Read`、`Write`、`Glob`、`Grep`、`WebSearch` 工具
5. AI Agent 具备自己的循环，能够在一次用户请求内连续思考、调用工具、产出结果，而不是只执行单步改写

## 产品决策

- 右侧栏主形态从“选区改写面板”升级为“聊天优先，改写是其中一种能力”
- 选区仍然保留，但从必填输入降级为可选上下文
- AI Agent 的默认工作对象是当前 workspace，而不是仅当前选中文本
- AI Agent 工具调用应显式可观察，至少展示当前步骤、工具名和阶段性结果
- `Write` 工具默认写入 backend workspace，而不是由前端直接拼接富文本结果
- Agent loop 需要有明确的终止条件、步数上限和错误收敛策略，避免无限循环
- `WebSearch` 属于受控工具，默认用于补充事实和外部资料，不应绕过当前文档上下文直接重写整篇内容

## 范围拆分

### 第一阶段：聊天壳层可用

- 输入框支持回车发送，`Shift+Enter` 换行
- 支持多轮消息列表
- assistant 消息在 agent 运行过程中持续流式更新，而不是结束后一次性出现
- 新消息、流式输出和工具过程自动滚动到底部
- 支持基础的用户消息 / assistant 消息 / 系统状态消息区分

### 第二阶段：从选区改写切到任务型 Agent

- 用户无选区也可提交指令
- 请求体支持携带当前文档、可选选区、当前路径、历史消息
- AI 可根据任务决定是回答、读取、搜索、改写还是写回
- 保留“审核后应用”与“直接写入”的策略开关，避免所有任务都变成隐式落盘

### 第三阶段：工具与执行循环

- 增加 `Read`、`Write`、`Glob`、`Grep`、`WebSearch` 工具
- 将工具调用接入统一的 agent runtime，而不是散落在 route 或 service 中手工拼接
- 支持单次请求内多步调用与阶段性事件流输出
- 记录每一步的输入摘要、工具结果摘要和最终结论

## 主要实现方向

### 前端

- 重构右侧栏状态模型，从“单 run 候选结果”扩展为“会话 + 消息 + 运行步骤”
- 输入组件改为聊天输入模型，支持回车发送与流式禁用态
- 消息区实现自动滚动，但在用户主动上滑查看历史时不强制打断
- assistant 消息需要支持 token 级或 chunk 级增量更新，避免整条消息在结束后才渲染
- UI 需要同时承载纯文本答复、工具调用日志、候选改写结果和写入确认状态

### 后端

- 新增面向会话式 agent 的 API，而不是继续复用单用途 rewrite 接口硬撑
- 将当前 workspace service 暴露给 agent tools，作为读写和检索的真相源
- 基于 LangChain 组织工具与循环执行，而不是单次 prompt -> 单次结果
- 事件流需要覆盖消息增量、工具调用开始/结束、写入结果、最终完成态，不能只在 agent 完成后下发最终消息

## 验收口径

1. 用户在右侧栏输入一条消息，按回车即可发送，`Shift+Enter` 不会误发送
2. 连续发送多条消息后，右侧栏能保留会话上下文并生成相关连续回复
3. assistant 回复在生成过程中持续更新到 UI，而不是等整个 run 完成后一次性显示
4. 即使没有选中正文，用户也可以要求 AI 总结、查找、修改或补写当前文档
5. 当任务需要读取、搜索或修改时，UI 能展示 agent 使用了哪些工具
6. 对于一次复杂请求，agent 可以在单次 run 中执行多步工具调用后再给出最终结果
7. 自动滚动在正常聊天时可用，查看历史时不会造成明显打断

## 风险与约束

- 当前前端状态管理主要围绕 rewrite candidate 设计，直接叠加 chat 容易形成双轨状态混乱
- `Write` 工具与富文本编辑器缓冲之间需要重新定义同步边界，否则容易覆盖用户本地未保存编辑
- `WebSearch` 会引入外部信息与时效性问题，必须保留来源与调用边界
- agent loop 如果没有步数限制、超时限制和取消机制，会显著放大成本与失败面

## 当前建议顺序

1. 先完成右侧栏聊天壳层和多轮状态模型
2. 再把“无选区也可发起任务”接入后端新接口
3. 然后接 `Read` / `Glob` / `Grep`
4. 最后接 `Write` / `WebSearch` 和完整 agent loop 观测
