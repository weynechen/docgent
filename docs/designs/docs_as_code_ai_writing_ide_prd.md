# Docs-as-Code AI Writing IDE 产品需求文档（PRD）

## 1. 文档信息

- 产品代号：Writing IDE
- 产品定位：面向高频写作者的、以 Docs-as-Code 为核心理念的 Markdown-first、Version-native、AI-native 写作编辑器
- 当前阶段：MVP 规划
- 技术方向：React + Tiptap 为 MVP 核心；后续可扩展 Electron、本地文件与 Git 作为底层实现
- 核心目标：让用户在同一个编辑器内完成“写作、AI 改写、版本管理、平台优化、导出发布”的完整闭环

---

## 2. 背景与问题定义

### 2.1 当前用户工作流

目标用户目前的工作流通常如下：

1. 自己记录灵感或写下大纲
2. 快速写出未经整理的原始表达
3. 将文本复制到网页 AI 工具中做润色或改写
4. 使用面向特定平台的检查提示词做内容审查
5. 根据审查结果继续人工修改，或局部让 AI 协助修改
6. 用 Git 或手工副本管理版本
7. 最终手动整理格式，再发布到不同平台

### 2.2 当前问题

该流程存在以下问题：

#### 2.2.1 上下文切换严重
用户需要在 Obsidian、网页 AI、发布平台之间频繁来回切换，打断写作思路。

#### 2.2.2 AI 不在写作主流程内
AI 只是外置助手，而不是编辑器内部的一等能力。用户需要复制、粘贴、对比、手工替换。

#### 2.2.3 本地文件与 AI 工作流割裂
Obsidian 适合本地 Markdown 管理，但 AI 能力弱；网页 AI 能做润色，但脱离本地文件与 Git 仓库。

#### 2.2.4 缺少结构化版本管理
写作过程本质上有阶段性演进，但大多数写作工具没有把版本、diff、提交、回滚作为核心设计。

#### 2.2.5 缺少平台适配闭环
针对知乎、Twitter 等平台的内容优化和格式整理依赖外部提示词和手工操作，效率低且不稳定。

### 2.3 产品机会

存在一个明显空白：

> 将 Docs-as-Code 的理念引入写作工具，把文档结构化、可演进、可版本回溯、可审查修改的能力统一到一个写作 IDE 中。

该产品不是普通笔记应用，也不是普通 AI 聊天工具，而是：

> Docs-as-Code AI Writing IDE

这里的 Docs-as-Code 指的是：

- 文档有明确结构与生命周期
- 修改是可审查、可回退、可对比的
- 写作过程像构建文档系统，而不是一次性生成文本
- 版本管理是核心能力，但不要求用户理解 Git

---

## 3. 产品目标

### 3.1 总体目标

打造一款围绕 Docs-as-Code 理念工作的 AI 写作编辑器，使用户可以像构建文档系统一样写作、改写、审查和回溯版本。

### 3.2 核心目标

1. 用户在一个界面内完成写作与 AI 协作
2. 文档修改天然具备版本化、可对比、可回退能力
3. AI 修改以“建议 + 应用”的方式介入，而非黑盒替换
4. 用户感知到的是清晰的版本管理，而不是底层 Git 操作
5. 后续可扩展到本地文件、平台适配、导出与发布

### 3.3 MVP 目标

在 MVP 阶段，优先验证以下命题：

> 用户是否愿意在一个 Docs-as-Code 风格的写作界面中，通过选区对话、AI 改写、建议应用和版本回溯完成主要写作工作。

只要该命题成立，即证明产品核心价值成立。

## 4. 产品原则

### 4.1 Docs-as-Code First
文档不是一次性输入框内容，而是具备结构、阶段、版本和演进过程的工作对象。

### 4.2 Markdown-first
文档核心表达应兼容 Markdown 语义，便于后续导出、平台适配和长期维护。

### 4.3 Version-native
版本管理是产品一等能力。用户应能自然地创建阶段版本、查看变更、回溯历史，但不必直接接触 Git 概念。

### 4.4 AI-native
AI 是编辑器内部能力，而不是外部网页工具。

### 4.5 Human-in-control
AI 只提出建议，不应无提示地篡改整篇文档；应用修改必须可预览、可拒绝、可撤销。

### 4.6 Implementation-Decoupled
Docs-as-Code 是产品理念；Local-first、Electron、Git 是可选实现路径，而不是 MVP 的产品前提。

## 5. 目标用户

### 5.1 核心用户画像

#### 用户类型 A：高频技术写作者
- 经常写博客、技术文章、产品思考、长推文
- 习惯 Markdown 或结构化写作
- 希望写作过程具备清晰阶段与版本

#### 用户类型 B：AI 增强型独立创作者
- 先自己写，再让 AI 协助润色或审查
- 需要对不同平台做定向优化
- 不希望在多个工具中来回切换

#### 用户类型 C：重视可回溯写作过程的用户
- 希望知道每次修改发生了什么
- 希望能回到任意历史版本
- 不一定懂 Git，但需要版本化心智

### 5.2 非目标用户

- 只需要轻量记笔记、没有高频写作需求的用户
- 完全不接受本地文件或 Git 概念的纯消费型用户
- 只想聊天生成整篇文章的用户

---

## 6. 用户核心场景

### 6.1 场景一：快速起草 + 局部润色
用户打开本地仓库中的一篇 Markdown 文档，快速写下原始内容。选中一段表达不顺的文字，在右侧输入“更清晰一些，保留我原来的判断”，AI 返回建议版本，用户查看 diff 后点击应用。

### 6.2 场景二：一边写一边创建阶段版本
用户完成文章结构后保存一个版本；完成前半部分草稿后再保存一个版本；AI 优化完后再保存一个版本。整个写作过程具备清晰里程碑，用户可以随时回到任意历史状态。

### 6.3 场景三：平台定向优化
用户完成文章后，切换到“知乎优化”模式，AI 给出不符合平台风格的段落建议。用户逐条接受或忽略。

### 6.4 场景四：导出发布版本
用户将同一篇原稿导出为知乎版和 Twitter thread 版，减少重复改写成本。

## 7. 产品范围

### 7.1 MVP 范围（第 1 阶段）

必须包含：

1. Tiptap 编辑界面
2. 三栏式布局
3. 中栏顶部标签栏，展示当前打开文件名
4. 左右侧栏宽度可调
5. 左右侧栏支持折叠与从标签栏两侧恢复
6. 选区对话
7. AI 改写建议
8. 建议预览与应用（accept / reject）
9. 基础版本管理能力（创建版本、查看最近版本、回到上一个版本）

MVP 可选但建议尽早抽象：

10. 文档读取与写入接口层
11. diff 组件基础能力

### 7.2 第 2 阶段范围

1. Electron 桌面化
2. 本地目录打开
3. 本地 Markdown 文件管理
4. 文件树
5. 底层 Git 集成或等价版本存储实现

### 7.3 第 3 阶段范围

1. 平台优化规则
2. AI 审查与建议 UI
3. 导出格式转换
4. 一键发布能力（视平台开放能力决定）

### 7.4 明确不在 MVP 范围

1. 云同步
2. 多人协作
3. 插件市场
4. 双链知识图谱
5. 全自动写完整篇内容的 agent 系统
6. 完整的发布自动化到所有平台

---

## 8. 核心价值主张

### 8.1 对用户的价值

- 写作不再被 AI 工具打断
- 修改过程更可控
- 文档真实存在于本地
- 写作过程可版本化管理
- 从草稿到发布形成统一闭环

### 8.2 与现有工具的差异

#### 相比 Obsidian
- 更强的 AI 写作交互
- 面向“改稿”而不是单纯“记笔记”
- 有 Cursor 风格的建议应用体验

#### 相比网页 AI
- 不脱离本地文件与仓库
- 支持上下文内修改而不是复制粘贴
- 能形成长期写作工作流

#### 相比 Notion 类编辑器
- 内容不是云数据库主导
- 支持 Git-native 工作模式
- 更适合技术型、工程型写作者

## 9. 信息架构

### 9.1 三栏结构

#### 左栏：文件与项目区
职责：管理仓库、目录树、文件列表与文档基础操作

包含：
- 打开项目 / 仓库
- 文件树
- 新建文件
- 重命名文件
- 删除文件
- 当前文件状态提示（未保存 / 已修改）
- 侧栏宽度拖拽
- 侧栏折叠按钮

#### 中栏：写作编辑区
职责：承载正文编辑、选区操作、内容展示与修改落地

包含：
- 顶部标签栏（展示当前打开文件）
- 文档标题
- Markdown 编辑区
- 当前选区高亮
- 建议应用反馈
- 撤销 / 重做
- 左右侧栏恢复入口
- 底部轻量状态栏（保存状态、字数、字符数）

#### 右栏：AI 协作区
职责：承载当前选区上下文、对话与变更建议

包含：
- 当前选区对应消息
- 指令输入框
- AI 输出建议
- 建议卡片
- accept / reject / retry
- 侧栏宽度拖拽
- 侧栏折叠按钮

---

## 10. 用户流程设计

### 10.1 MVP 主流程

#### 流程：选区改写
1. 用户打开文档
2. 用户在中栏选中一段文本
3. 右栏自动显示“当前选区”
4. 用户点击快捷动作，或输入自由指令
5. 系统将选区文本与必要上下文发送给 AI
6. AI 返回建议文本与可选解释
7. 右栏显示 old/new 对比
8. 用户点击 accept 或 reject
9. 如果 accept，则中栏替换原选区内容
10. 用户可继续写作，或撤销本次改动

### 10.2 阶段二流程：本地项目打开
1. 用户打开应用
2. 用户选择一个本地目录作为项目
3. 左栏渲染目录树
4. 用户点击 `.md` 文件
5. 系统读取本地文件内容
6. 中栏加载内容并进入编辑状态
7. 用户保存后系统写回本地磁盘

### 10.3 阶段三流程：Git 提交
1. 用户完成某阶段写作
2. 点击“提交版本”
3. 系统显示已修改文件与摘要
4. 用户输入 commit message
5. 系统执行本地 Git 提交
6. 左栏或状态区显示当前版本状态

### 10.4 阶段三流程：平台优化
1. 用户完成文章主稿
2. 选择目标平台，如知乎
3. 系统基于平台规则对全文或段落进行分析
4. 右栏展示建议列表
5. 用户逐项接受、忽略或再次生成
6. 生成目标平台版本

## 11. 功能需求

## 11.1 编辑器模块

### 功能目标
提供稳定的富文本编辑体验，并可与 Markdown 形态可靠互转。

### 需求点
1. 支持基础段落、标题、加粗、斜体、列表、引用、代码块等基础能力
2. 支持选区获取与定位
3. 支持中栏内容的程序化替换
4. 支持撤销 / 重做
5. 支持内容变化监听
6. 支持初始化内容加载
7. 支持将编辑状态导出为 Markdown
8. 支持从 Markdown 加载为编辑器内容

### 设计要求
- 初版可优先保证常见 Markdown 语法正确
- 避免过早支持复杂块结构导致 Markdown 往返失真
- 对 AI 修改的应用必须能精准定位替换范围

---

## 11.2 三栏布局模块

### 功能目标
提供清晰稳定的工作台结构，让文件、写作和 AI 协作并行。

### 需求点
1. 左、中、右栏固定可视
2. 支持调整栏宽
3. 在中等及以上屏幕下默认展示三栏
4. 右栏在无选区时仍可做通用对话，但主能力聚焦于“对当前选区操作”
5. 支持栏位折叠
6. 中栏顶部提供标签栏，展示当前打开文档
7. 左右栏折叠后，恢复入口放在标签栏左右两侧

### 设计要求
- 中栏必须是视觉中心
- 右栏不应喧宾夺主，不做完整 IM 风格聊天界面
- 左栏优先强调文档仓库感，而非知识图谱感
- 标签栏负责体现“当前工作文件”语义，不扩展为复杂浏览器式多标签系统

---

## 11.3 选区上下文模块

### 功能目标
让系统知道“用户当前想改哪一段”，并将其变成 AI 协作的最小单位。

### 需求点
1. 用户选择文本后，系统获取选区文本
2. 系统能记录选区在当前文档中的范围
3. 右栏展示选区摘要
4. 若选区变化，右栏状态同步更新
5. 若内容变化导致选区失效，应提示用户重新选择

### 设计要求
- 不强依赖复杂 AST patch，初版允许以“当前选区直接替换”为主
- 后续可引入更稳定的 range mapping 或基于 transaction 的位置同步

---

## 11.4 AI 指令改写模块

### 功能目标
实现用户对选区的 AI 定向改写。

### 需求点
1. 支持用户输入自由指令
2. 支持从编辑器工具条直接触发默认改写
3. 请求参数至少包含：选区文本、用户指令、必要上下文
4. AI 返回建议文本
5. UI 显示建议文本及解释（可选）
6. 支持 retry

### 输入结构建议
- selectedText
- instruction
- documentTitle
- beforeText（可选）
- afterText（可选）
- writingGoal（可选）
- targetPlatform（可选）
- styleProfile（可选）

### 输出结构建议
- suggestedText
- explanation（可选）
- confidence（可选，后续）

### 设计要求
- 第一版不做复杂 agent，自由指令 + 改写已经足够验证价值
- prompt 设计必须强调“保留原意”“不要擅自扩写事实”“尽量不改变作者判断”
- 右栏不默认展示固定快捷按钮组，避免与正文竞争注意力

---

## 11.5 建议预览与应用模块

### 功能目标
让 AI 修改以“可审查补丁”的方式进入文档。

### 需求点
1. 显示原文与建议版本
2. 支持 diff 视图或轻量建议卡片
3. 支持 accept
4. 支持 reject
5. 支持 apply 后高亮反馈
6. 支持一次建议只作用于当前选区
7. 支持撤销最近一次应用

### 设计要求
- accept / reject 是核心体验，不可省略
- UI 风格应接近 Cursor 的代码建议感，而不是一次性整段覆盖
- 建议应用必须稳定，不能破坏文档其他位置

---

## 11.6 版本管理模块

### 功能目标
让写作过程具备自然的阶段版本、变更对比与历史回溯能力。

### 用户侧概念
用户看到的是：
- 保存版本
- 查看版本历史
- 比较当前与历史差异
- 回到某个历史版本

用户不需要理解：
- Git commit
- branch
- rebase
- stash

### 需求点
1. 支持手动创建版本快照
2. 支持自动记录关键 AI 修改节点（后续增强）
3. 支持查看最近版本列表
4. 支持查看某版本与当前版本的 diff
5. 支持恢复到指定历史版本
6. 支持版本命名或备注（后续增强）

### 设计要求
- MVP 可先实现轻量版本存储机制
- 底层可使用 Git，也可先用应用内快照实现
- 用户界面统一使用“版本”而不是“提交”

## 11.7 本地 Markdown 文件模块

### 功能目标
将文档真实读写到用户本地目录中。

### 需求点
1. 选择项目目录
2. 递归读取 Markdown 文件
3. 左栏展示文件树
4. 点击文件加载内容
5. 保存时写回磁盘
6. 监听外部文件变化（后续增强）

### 设计要求
- 只把 `.md` 作为 MVP 主文件类型
- 防止误覆盖，保存策略需明确
- 未来可支持 frontmatter，但 MVP 可先不深做

---

## 11.8 底层存储与 Git 集成模块（内部实现）

### 功能目标
为后续 Docs-as-Code 能力提供可靠的底层实现，包括本地文件与版本化存储。

### 说明
该模块属于实现层，不直接暴露给用户。用户侧只感知“文档”“版本”“回溯”“差异”。

### 需求点
1. 支持将版本操作映射到底层存储
2. 支持文件级 diff 能力
3. 支持持久化版本历史
4. 后续支持将版本系统落到 Git 仓库

### 设计要求
- MVP 不强依赖 Git 直接暴露
- 可先用应用内版本快照，后续切换或兼容 Git
- UI 文案避免出现 Git 专有术语

## 11.9 平台优化模块

### 功能目标
针对不同博客平台生成优化建议，而不是直接强行改写全文。

### 需求点
1. 选择目标平台
2. 调用平台规则 prompt 分析当前内容
3. 输出问题列表与改进建议
4. 支持逐条应用修改
5. 支持生成平台适配版本副本

### 设计要求
- 先从知乎、Twitter 两个平台开始
- 初版以“规则检查 + AI 建议”为主，不做完全自动发布
- 建议展示风格应与 AI 改写模块一致，保持 accept / reject 统一体验

---

## 11.10 平台导出模块

### 功能目标
将 Markdown 或文档内容导出为适合目标平台的文本结构。

### 需求点
1. 导出为标准 Markdown
2. 导出为纯文本
3. 导出为适合 Twitter Thread 的段落序列
4. 导出为知乎可粘贴格式（后续可增强）

### 设计要求
- MVP 后置，不阻塞核心价值验证
- 一键上传依赖平台开放能力，需逐平台评估

---

## 12. 非功能需求

### 12.1 性能
- 打开普通 Markdown 文档应流畅
- 选区改写响应需有清晰 loading 状态
- UI 操作不能因 AI 请求而整体卡死

### 12.2 稳定性
- 文档保存必须可靠
- AI 应用错误不能污染原文
- 修改建议必须具备撤销能力

### 12.3 可维护性
- 编辑器层、文档层、AI 层、Git 层要清晰分离
- 数据接口优先抽象，不与 Electron API 强耦合

### 12.4 可扩展性
- 后续可新增平台适配器
- 后续可新增 AI 操作类型
- 后续可新增 Git 能力

### 12.5 隐私
- 本地文件默认不上传云端存储
- 仅在用户发起 AI 请求时上传相关文本
- 应明确提示哪些内容会发送到模型服务

## 13. 交互设计细化

## 13.1 中栏与右栏联动

### 规则
1. 无选区时：右栏显示通用助手或提示用户选择文本
2. 有选区时：右栏进入“针对当前选区”模式
3. 当前选区以对话消息形式呈现，而不是固定信息卡片
4. 用户切换选区时，当前未应用建议应提示是否保留或丢弃

## 13.2 AI 建议卡片

每条建议卡片包括：
- 指令摘要
- 原文片段
- 建议版本
- diff 视图切换（可选）
- accept / reject / retry

## 13.3 应用反馈

accept 后：
- 中栏目标区域高亮闪烁一次
- 右栏标记为已应用
- 底部状态栏记录一次“AI 修改已应用”事件

## 13.5 工作区框架约束

### 规则
1. 底部状态栏应尽量轻薄，避免占据正文垂直空间
2. 保存提示、字数、字符数、最近一次 AI 应用状态统一收敛到状态栏
3. 左右侧栏折叠后，主编辑区宽度应立即扩张
4. 折叠恢复按钮应固定出现在标签栏左右两侧，避免漂浮在正文上

## 13.4 错误状态

需要处理：
- 无网络
- 模型返回为空
- 选区失效
- 应用失败
- 文件保存失败

## 17. MVP 验收标准

满足以下条件，可认为核心 MVP 成立：

1. 可以打开一个文档进行编辑
2. 用户可以选中任意一段文本
3. 右侧能自动识别当前选区
4. 用户可以输入修改要求
5. AI 能返回建议版本
6. 系统能展示原文与建议的对比
7. 用户可以 accept 或 reject
8. accept 后仅替换目标选区内容
9. 替换后文档内容保持结构稳定
10. 支持撤销最近一次应用

## 18. 风险与关键难点

### 18.1 Markdown 与富文本状态一致性
难点：Tiptap 编辑态与 Markdown 文件态的双向转换可能出现丢失或偏差。

策略：
- MVP 只支持核心 Markdown 语法
- 尽早做往返测试
- 避免在早期引入复杂块对象

### 18.2 AI 建议的安全应用
难点：如何准确把 AI 建议落到正确位置，且不影响其他内容。

策略：
- 初版只允许对当前选区做替换
- 应用前必须预览
- 提供撤销与错误回退

### 18.3 本地文件与外部修改冲突
难点：文件可能被用户在外部编辑器中同时修改。

策略：
- 初版可先弱化冲突处理
- 后续增加文件变化监听与冲突提示

### 18.4 过早引入复杂 agent
难点：过早做自动多轮 agent 会拖慢开发，且不利于验证核心价值。

策略：
- MVP 只做 instruction-based rewrite
- 平台优化和全篇审查放到后续阶段
